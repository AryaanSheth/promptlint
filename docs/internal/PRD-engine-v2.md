### Bug 1.1 ŌĆö Injection rule stops after first match

**File:** `cli/promptlint/rules/security.py:95`
**Severity:** Correctness bug

```python

    ...
    results.append(...)
    break  # ŌåÉ stops here. second injection attempt is never reported.

A prompt with two injection payloads only reports one. Remove the `break`. The loop should collect every matching pattern and append a result for each. The fix pipeline (`_remove_injection_content`) already handles multiple lines ŌĆö only the detection is broken.

**Fix:** Remove `break` at line 95. Change the single-match `if match` block to append all matches across all patterns.

### Bug 1.2 ŌĆö Obfuscation detection flag is always wrong

**File:** `cli/promptlint/rules/security.py:82`
**Severity:** Correctness bug

```python
match = re.search(pattern, text, re.IGNORECASE)
if not match:
    match = re.search(pattern, normalized, re.IGNORECASE)
...
obfuscated = match.string != text  # ŌåÉ compares Python string objects
```

`match.string` is the string that was passed to `re.search`. If the first search (on `text`) finds a match, `match.string is text` is `True`, so `obfuscated = False` ŌĆö correct. But if the first search fails and the second search (on `normalized`) finds a match, `match.string is normalized` ŌĆö but the comparison `match.string != text` compares by identity, and since `normalized` is a different object, this should work. However: if both `text` and `normalized` match the same pattern, the first branch always wins and `obfuscated` is always `False` even when the attack used leetspeak that happens to also match the raw pattern.

The correct approach: track explicitly which branch produced the match.

```python
match = re.search(pattern, text, re.IGNORECASE)
from_normalized = False
if not match:
    match = re.search(pattern, normalized, re.IGNORECASE)
    from_normalized = True
...
obfuscated = from_normalized
```

### Bug 1.3 ŌĆö `clarity-vague-terms` fires on legitimate prompts

**File:** `cli/promptlint/rules/quality.py:102`
**Severity:** High false-positive rate ŌåÆ user churn

```python
(r"\b(good|bad|nice|better|best)\b", "subjective term without criteria"),
(r"\b(appropriate|suitable|relevant|proper)\b", "undefined standard"),
```

These patterns fire WARN on:
- "Return the **best** option" (legitimate constraint)
- "Follow **best** practices" (legitimate instruction)
- "A **good** example would be" (legitimate guidance)
- "Return **appropriate** error messages" (legitimate instruction)
- "Use the most **suitable** format" (legitimate instruction)

These are not vague in the context of LLM prompts. They are directional instructions. The vague terms that actually degrade LLM output quality are: `something`, `somehow`, `various`, `several`, `etc`, `and so on`, `stuff`, `things`.

**Fix:** Remove the `good/bad/nice/better/best` pattern entirely. Remove `appropriate/suitable/relevant/proper`. Keep only:
```python
vague_patterns = [
    (r"\b(some|several|various|many|few)\b", "vague quantifier ŌĆö specify a number or range"),
    (r"\b(stuff|things|etc|and so on|and more)\b", "trailing vague catch-all ŌĆö enumerate explicitly"),
    (r"\b(somehow|somewhere|sometime)\b", "unspecified manner/place/time"),
    (r"\bmaybe\b(?!\s+(?:null|undefined|none|empty))", "uncertain language in an instruction context"),
]
```

The `maybe` exclusion prevents false positives on code-adjacent prompts that reference `maybe null`.

### Bug 1.4 ŌĆö Sentence length splitting breaks on abbreviations and decimals

**File:** `cli/promptlint/rules/quality.py:162`
**Severity:** Medium false-positive rate

```python
sentences = re.split(r'[.!?]+', text)
```

This splits on every period, including:
- `e.g.` ŌåÆ splits into `"e"`, `"g"`, rest of sentence
- `i.e.` ŌåÆ same
- `v1.0` ŌåÆ splits at decimal
- `3.14` ŌåÆ splits at decimal
- URLs ŌåÆ `https://example.com/path` ŌåÆ splits on nothing but still mangled
- `Mr. Smith` ŌåÆ splits after `Mr`

Results in spurious short "sentences" being created and legitimate long sentences being missed.

**Fix:** Replace with a smarter split that excludes known abbreviation patterns:

```python
# Split on sentence-ending punctuation NOT followed by a lowercase letter or digit
# (abbreviations like e.g. are always followed by lowercase or a space+lowercase)
sentences = re.split(r'(?<=[^A-Z][.!?])\s+(?=[A-Z])|(?<=[.!?]{2})\s+', text)
```

Or more robustly: use the `nltk.sent_tokenize` if available, fall back to the current regex only as a last resort. Add `nltk` as an optional dependency.

### Bug 1.5 ŌĆö `actionability-weak-verbs` doesn't check weak verbs

**File:** `cli/promptlint/rules/quality.py:208ŌĆō225`
**Severity:** Rule is misnamed and misimplemented

### Rule 2.1 ŌĆö `jailbreak-pattern` | CRITICAL | Security

**Category:** Security
**Severity:** CRITICAL
**Fixable:** Yes (remove offending lines, same as injection fix)
**Files to create/modify:** `security.py`, `registry.py`, `config.py`

**What it detects:** Social-engineering patterns that use roleplay, hypotheticals, or fictional framing to bypass AI guardrails. Distinct from `prompt-injection` (which targets your system prompt) ŌĆö jailbreaks target your deployed chatbot through user input.

**Why it matters:** Enterprise security teams treat jailbreaks and injection as separate risk vectors. They want separate reports for each. Having both as distinct rules lets you map to OWASP LLM Top 10 (LLM01: Prompt Injection, which covers both).

**Default patterns:**
```python
JAILBREAK_PATTERNS = [
    r"\b(?:you are now|pretend you are|act as if you are)\b",
    r"\b(?:DAN|do anything now)\b",
    r"\b(?:ignore your|forget your|disregard your)\s+(?:training|guidelines|instructions|rules|restrictions)",
    r"\b(?:for a (?:story|book|novel|game|roleplay|hypothetical))\b",
    r"\bhypothetically(?:\s+speaking)?,?\s+(?:if|suppose|imagine)",
    r"\b(?:no restrictions|no limits|no rules|no filters|no guidelines)\b",
    r"\b(?:you have no|you don't have|you no longer have)\s+(?:restrictions|filters|guidelines|rules)",
    r"\bjailbreak\b",
    r"\b(?:developer mode|god mode|unrestricted mode|admin mode)\b",
    r"(?:ignore|forget|disregard)\s+(?:all\s+)?(?:previous|prior|above|earlier)\s+(?:instructions|prompts|context|conversation)",
]
```

Apply the same leet-speak and unicode normalization from `_normalize_for_matching`.

**Config:**
```yaml
rules:
  jailbreak_pattern:
    enabled: true
    patterns:   # user can extend
      - "custom pattern here"
```

**Registry entry:** Add to `registry.py` with:
- `id`: `"jailbreak-pattern"`
- `category`: `"Security"`
- `default_severity`: `"CRITICAL"`
- `fixable`: `True`

### Rule 2.2 ŌĆö `role-clarity` | WARN | Quality

**Category:** Quality: Structure
**Severity:** WARN
**Fixable:** No (requires human judgment to write a good persona)
**Files:** `quality.py`, `registry.py`

**What it detects:** System prompts that lack a clear persona or role definition. Prompts without role definition produce inconsistent, generic, often lower-quality outputs because the model defaults to a generic assistant persona.

**Detection logic:**
```python
def check_role_clarity(text: str, config: PromptlintConfig) -> List[Dict]:
    if not config.enabled_rules.get("role-clarity", True):
        return []

    # Only fire on longer prompts ŌĆö short prompts often don't need a persona
    if len(text.split()) < 30:
        return []

    # Check if prompt looks like a system prompt (instructional tone)
    is_instructional = bool(re.search(
        r"\b(you|your|respond|answer|help|assist|always|never|must|should)\b",
        text, re.IGNORECASE
    ))
    if not is_instructional:
        return []

    # Check for role definition
    has_role = bool(re.search(
        r"\b(you are|you're|act as|your role is|you will serve as|"
        r"you are a|you are an|you work as|your job is|your task is|"
        r"you are responsible for|you specialize in)\b",
        text, re.IGNORECASE
    ))

    if not has_role:
        return [{
            "level": "WARN",
            "rule": "role-clarity",
            "message": "No role or persona defined. Add 'You are a [role]...' to improve output consistency.",
            "line": "-",
            "context": _preview(text, config.preview_length),
        }]
    return []
```

### Rule 2.3 ŌĆö `output-format-missing` | WARN | Quality

**Category:** Quality: Completeness
**Severity:** WARN
**Fixable:** No
**Files:** `quality.py`, `registry.py`

**What it detects:** Prompts that ask for structured output (lists, data, comparisons) but don't specify the output format. This is the #1 cause of inconsistent LLM outputs in production ŌĆö the model invents its own format on every call.

**Detection logic:**
```python
OUTPUT_INSTRUCTION_VERBS = r"\b(list|extract|find|return|give me|provide|output|generate|create|show|summarize|identify|enumerate)\b"

OUTPUT_FORMAT_KEYWORDS = r"\b(JSON|XML|CSV|markdown|bullet|numbered|table|plain text|HTML|YAML|format:|return as|output as|structured as|schema)\b"

def check_output_format(text: str, config: PromptlintConfig) -> List[Dict]:
    if not config.enabled_rules.get("output-format-missing", True):
        return []
    if len(text) < 60:
        return []

    has_output_instruction = bool(re.search(OUTPUT_INSTRUCTION_VERBS, text, re.IGNORECASE))
    has_format_spec = bool(re.search(OUTPUT_FORMAT_KEYWORDS, text, re.IGNORECASE))

    if has_output_instruction and not has_format_spec:
        return [{
            "level": "WARN",
            "rule": "output-format-missing",
            "message": "Output instruction detected but no format specified (JSON, markdown, bullet list, etc.). Unspecified format produces inconsistent results.",
            "line": "-",
            "context": _preview(text, config.preview_length),
        }]
    return []
```

### Rule 2.4 ŌĆö `pii-in-prompt` | WARN | Security + Compliance

**Category:** Security
**Severity:** WARN
**Fixable:** No (can't auto-redact ŌĆö human must verify)
**Files:** `security.py`, `registry.py`, `config.py`

**What it detects:** Personally Identifiable Information hardcoded in prompt files. Engineers routinely paste real data when prototyping and accidentally commit it. This is a GDPR/HIPAA blocker for enterprise adoption.

PII_PATTERNS = [
    (r"\b[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Z|a-z]{2,}\b", "email address"),

    (r"\b5[1-5][0-9]{14}\b", "Mastercard number pattern"),
    (r"\b(?:25[0-5]|2[0-4]\d|[01]?\d\d?)\.(?:25[0-5]|2[0-4]\d|[01]?\d\d?)\.(?:25[0-5]|2[0-4]\d|[01]?\d\d?)\.(?:25[0-5]|2[0-4]\d|[01]?\d\d?)\b", "IP address"),

SECRET_PATTERNS = [

### Rule 2.6 ŌĆö `hallucination-risk` | WARN | Quality

**Category:** Quality: Reliability
**Severity:** WARN
**Fixable:** No
**Files:** `quality.py`, `registry.py`

**What it detects:** Prompts that ask the model to state current facts, real-world data, or time-sensitive information without providing grounding context. These prompts will produce hallucinations consistently.

**Detection logic:**
```python
FACTUAL_QUESTION_PATTERNS = [
    r"\b(what is|what are|who is|who are|when did|when was|where is|how many|how much)\b",
    r"\b(current(ly)?|latest|recent(ly)?|today|now|as of|up to date)\b",
    r"\b(tell me about|give me (the|a) (?:list|summary|overview) of)\b",
]

GROUNDING_INDICATORS = [
    r"\{[\w\s]+\}",          # template variable like {context} or {documents}
    r"<context>",            # XML context tag
    r"Context:",             # heading-style context
    r"```",                  # code block (often contains data)
    r"Given the following",
    r"Based on the (?:following|above|provided)",
    r"Using the (?:data|information|context|text) (?:below|above|provided)",
]

def check_hallucination_risk(text: str, config: PromptlintConfig) -> List[Dict]:
    if not config.enabled_rules.get("hallucination-risk", True):
        return []

    has_factual_request = any(
        re.search(p, text, re.IGNORECASE) for p in FACTUAL_QUESTION_PATTERNS
    )
    has_grounding = any(
        re.search(p, text, re.IGNORECASE) for p in GROUNDING_INDICATORS
    )

    if has_factual_request and not has_grounding:
        return [{
            "level": "WARN",
            "rule": "hallucination-risk",
            "message": "Prompt requests factual/current information without providing grounding context. Consider adding a {context} variable or <context> section with source data.",
            "line": "-",
            "context": _preview(text, config.preview_length),
        }]
    return []
```

### Rule 2.7 ŌĆö `context-injection-boundary` | WARN | Security

**Category:** Security
**Severity:** WARN
**Fixable:** No (requires architectural change to the prompt)
**Files:** `security.py`, `registry.py`

**What it detects:** Template variables (user-provided content) placed directly inside instructional text without a clear structural boundary. This is the architectural root cause of prompt injection ŌĆö when user content and system instructions share the same text context with no delimiter, injection attacks work by appending instructions to the user's input.

**Detection logic:**
```python
def check_injection_boundary(text: str, config: PromptlintConfig) -> List[Dict]:
    if not config.enabled_rules.get("context-injection-boundary", True):
        return []

    # Find template variables
    variables = re.findall(r'\{[\w\s]+\}|\{\{[\w\s]+\}\}', text)
    if not variables:
        return []

    results = []
    for var_match in re.finditer(r'\{[\w\s]+\}|\{\{[\w\s]+\}\}', text):
        # Check if this variable is inside a structural delimiter
        var_start = var_match.start()
        preceding_text = text[:var_start]
        following_text = text[var_start:]

        # Look for enclosing XML tags, markdown code fences, or section headers
        enclosed_by_xml = bool(
            re.search(r'<[\w_-]+>\s*$', preceding_text) and
            re.search(r'^\s*</[\w_-]+>', following_text)
        )
        enclosed_by_fence = bool(
            re.search(r'```\w*\s*$', preceding_text) and
            re.search(r'^\s*```', following_text)
        )
        preceded_by_header = bool(
            re.search(r'(?:User Input|User Message|Input|Query):\s*$', preceding_text, re.IGNORECASE)
        )

        if not (enclosed_by_xml or enclosed_by_fence or preceded_by_header):
            line_num = text.count('\n', 0, var_match.start()) + 1
            results.append({
                "level": "WARN",
                "rule": "context-injection-boundary",
                "message": f"Template variable '{var_match.group(0)}' is not enclosed by a structural delimiter. Wrap user input in XML tags or a labeled section to reduce injection risk.",
                "line": line_num,
                "context": _line_context(text, var_match.start(), config.context_width),
            })

    return results
```

### 3.1 ŌĆö Configurable rule severity

**Resolution:** After each Python rule fix in this PRD, port the change to the corresponding TypeScript file.
